# 知识沉淀原则与写作规范

撰写或修改 `knowledge/` 下的知识文档时，参考以下原则。

## 核心目的

知识沉淀**不是**将本次对话或修改的内容直接复述记录下来，而是为了在后续其他对话中更好地提示与指导 AI。始终以这个核心为出发点去整理和组织知识。

每次沉淀时应自问：这段知识在未来什么场景下会被 AI 需要？AI 读到它后能获得什么仅靠读代码得不到的认知？

## 沉淀原则

### 重点提取用户在对话中提供的背景知识

用户在对话中为了说明问题、描述需求而提到的背景信息，往往是代码中读不到的宝贵知识。这类信息是**最高优先级的沉淀目标**，包括但不限于：

- **业务场景描述**（哪些入口会携带什么参数、哪些角色可以访问什么功能）
- **用户行为路径**（用户从 A 到 B 再到 C 的操作链路）
- **产品规则和约束**（什么场景下应该做什么、不同身份的权限差异）
- **系统间的关联关系**（外部系统如何与当前项目交互）

这些信息通常散落在对话中，不会体现在代码注释或变量命名里，AI 仅靠读代码无法获知。沉淀时应将这些零散信息整理归纳到对应的知识文档中。

### 代码逻辑以外的知识优先

文档应聚焦于**代码之外的逻辑和知识**，而非代码本身的实现细节：

- **优先记录**：业务规则、设计决策的原因、模块间的隐含关系、产品约束
- **简要提及**：关键函数名或流程节点（作为索引线索即可）
- **避免展开**：通过阅读代码就能直接得知的具体实现逻辑、函数调用链、参数传递细节

判断标准：如果一段信息通过 `search_content` 搜索代码就能完整理解，那它不需要写入文档。文档要记录的是搜索代码也找不到答案的东西。

写入之前再多问一步：**这段知识有没有一个合适的代码位置可以承载？** 单个字段的枚举取值、某个函数的特殊行为、某段分支的触发条件，如果能简短地写在对应的类型定义、函数声明、分支语句旁边，就优先写在那里——代码注释和被描述的对象零距离，未来读代码的 AI 一定会看到，不会漏读也不会漂移。知识文档承载的是另一类知识：没有对应代码位置的（比如后台未声明枚举的字段取值、业务背景、产品约束），或跨了多个代码位置、任选一处注释都不完整的（比如由多个字段联合决定的业务流、模块间隐含的耦合关系）。

### 记录稳定的深层知识

优先记录不会随代码重构而失效的内容：

- **设计决策和原因**（为什么这样做，而不是具体怎么做）
- **易踩坑的模式**（什么情况下会出问题，问题的本质是什么）
- **概念和关系**（模块间的耦合关系、数据流向）

避免记录容易过时的内容：

- 具体的变量名、函数名、文件路径（这些通过读代码即可获知）
- 具体的代码片段（代码会变，逻辑描述更持久）
- 详细的字段生命周期表格（这些是代码的直接映射）

### 控制"本次修复"的比重

当因某次 bug 修复而沉淀知识时，修复本身只是引子。文档应聚焦于**背后的设计原理和通用模式**，而非详细描述本次改动。本次修复的具体做法可以作为"注意事项"简要提及，但不应占据文档主体。

### 挖掘 AI 不易直接推断的知识

文档的价值在于记录 AI 仅通过读代码**不容易直接理解到**的信息：

- 历史演变（为什么不用看起来更简单的方案）
- 隐含的约束（某些看似可选的逻辑实际上不能去掉）
- 跨模块的耦合关系（改 A 会影响 B，但代码结构上看不出来）

## 文档结构参考

```markdown
# [功能名称]

## 概述
功能作用和使用场景

## [按内容组织的主体章节]
根据要沉淀的知识自然组织，不要套固定模板

## 关键设计决策（可选）
当知识中确实包含经过权衡决策才产生的设计选择时需要此章节

## 注意事项
易踩坑的点、跨模块影响
```

## 交叉引用规范

知识文档之间应通过 Markdown 链接建立显式关联，帮助 AI 在阅读一篇文档时发现相关知识。

### 文档内引用

在文档正文中提到其他文档涉及的概念时，使用相对路径链接：

```markdown
详见 [Ticket 机制](../gift/core-concepts.md#ticket-机制)
```

### "相关文档"章节

当存在有实际关联的其他知识文档时，在文档末尾添加"相关文档"章节：

```markdown
## 相关文档

- [文档标题](相对路径) — 一句话说明关联关系
- [文档标题](相对路径) — 一句话说明关联关系
```

**写法要求**：
- 使用相对路径（`../module/doc.md`），不使用绝对路径
- 每条引用附带一句话说明**为什么相关**（不是复述目标文档内容）
- 只列出有实际关联的文档；没有则省略此章节，不要为"看起来完整"而凑弱关联

## 索引表"使用场景"列写法要求

`knowledge/README.md` 知识索引表中"使用场景"列：

- **只描述触发场景**：说明 AI 在什么情况下应该去读取该文档
- **不包含"怎么做"**：不要说明读取文档后应采取的具体操作
- **不复述文档内容**：如果使用场景描述把文档要点都说完了，文档就失去了存在意义
- **适当精简**：一句话概括即可
