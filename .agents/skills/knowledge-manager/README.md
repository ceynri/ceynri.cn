# knowledge-manager

一个与技术栈无关的 AI Agent Skill，用来在项目里搭建一个**能自我维护、自我进化、摩擦成本极低**的知识库，帮 AI 在后续会话中获得仅靠读代码得不到的业务上下文。

## 为什么需要它

项目里最有价值的知识往往代码里读不到——业务背景、做某个设计决策时权衡过的候选方案、后台不可见的字段取值、改 A 会悄悄影响 B 的耦合关系……这些信息散落在对话、会议、评审里，没有合适的载体留存。

传统 wiki 的办法是：开个文档、人工写、定期回顾、盼着别人维护。实际结果是文档永远过期、没人维护、AI 也不知道该去哪读。knowledge-manager 的出发点是**反过来**——把文档当作 AI 的长期记忆来设计，让文档的读者、维护者、检索者都是 AI 本身，人几乎不需要介入。

## 核心理念

### 对话即输入——把每次解释变成一次知识复利

人和 AI 协作时有一个反复发生的场景：AI 缺少某个业务背景，用户停下手上的事去解释一遍——"这个入口只有超管角色能进"、"这个字段后台是按订单状态动态下发的"、"这套方案之前试过 X，因为 Y 没走通才换的"。这些解释本身**就是最有价值的知识**：它们是代码里读不到的、AI 自己推不出来的、必须由真实业务上下文中的人才能给出的。

传统的协作模式里，这些解释说完就消散了——下次新会话、换个 AI 实例，用户又得从头讲一遍。knowledge-manager 把这些对话片段当成知识沉淀的**最高优先级目标**：每完成一次任务，skill 会从对话里识别哪些用户提供的背景信息值得留下，列出计划由用户确认后写入知识库。

效果是**复利**——每次对话都在给项目积累知识资产，越往后用户需要重复解释的东西就越少，AI 越来越多地"已经知道了"。这是 skill 跟传统文档工具最核心的差别：传统 wiki 是脱离工作流的额外负担，knowledge-manager 是从既有对话里顺手收割。

### 什么才是"知识"——代码读不到的才值得写

大多数"文档"失败，是因为它们在重复代码本身就能讲清楚的东西——函数调用链、参数传递、字段流转。这些一旦写下来就注定过期，且每次代码变动都要人盯着同步。

knowledge-manager 用一条简单的判断筛选内容：**`search_content` 能搜到的就不写**。文档只收纳代码里搜不到的知识——业务背景、设计决策的原因、产品约束、后台未声明枚举的字段取值、历史演变、隐含耦合。

再往下一层：**有合适代码位置可以承载的简短信息，优先写代码注释**——单字段枚举值、某函数的特殊行为、某分支的触发条件。代码注释和被描述对象零距离，AI 读代码一定会看到，不会漏读也不会漂移。知识文档承载的是另一类：**没有对应代码位置的**（业务背景），或**跨多个代码位置、任选一处注释都不完整的**（多字段联合决策流）。

这条双层过滤把文档规模降到了最小，也让每一条沉淀下来的内容都经得起时间考验。

### 如何让 AI 自动加载——索引驱动而非搜索驱动

一个知识库能不能真正生效，核心不在"写得好不好"，而在"AI 会不会读到"。knowledge-manager 的解法是：

- 首次安装时自动在项目的全局加载入口（`AGENTS.md` 或 always-apply rule）挂一段话，告诉 AI 在修改业务代码前先查 `knowledge/README.md` 索引
- 索引表用"使用场景"列描述每条知识的**触发条件**（AI 什么情况下该读这篇），而不是内容摘要
- AI 每次会话启动时读到入口挂钩 → 查索引 → 按场景判断需要读哪几篇 → 精准加载

这样 AI 从"需要人提醒"变成"自己会找"，且只读必要的文档，不污染上下文。

### 如何降低人工介入——全流程交给 skill

整个知识库生命周期几乎不需要人手动介入：

- **初始化**：首次调用时自动建 `knowledge/README.md`、挂全局入口，人不用配置
- **写入**：从 git diff 和当前对话自动识别可沉淀的内容，列出计划让用户确认后再写
- **索引同步**：文档与 `knowledge/README.md` 索引表的一致性由 skill 保证——不允许手动编辑 knowledge 目录，避免"改了文档忘了同步索引"这类沉默失败
- **归档**：文档过时走独立归档流程（移到同主题域的 `archive/` 子目录 + 顶部加过时声明），而非直接删除——历史决策仍可被追溯，不丢上下文
- **静默跳过**：判断没有可沉淀内容时不输出提示，避免给用户制造噪音

人类保留的唯一动作是在 skill 列出计划时说"可以"或"再改改"。

## 触发时机

- 完成较大功能开发后 / 对话中产生了代码外知识 / 用户明确说"记一下"
- Skill 也会在识别到可沉淀内容时主动提醒触发

## 安装后项目的样貌

```
项目根/
├── AGENTS.md                   # 或 always-apply rule，已挂钩知识库入口
└── knowledge/
    ├── README.md               # 知识索引表 + 归档表（skill 自动维护）
    ├── [模块名]/               # 按业务主题域组织
    │   ├── *.md
    │   └── archive/            # 已过时但保留供历史回查的文档
    ├── architecture/
    └── guides/
```

文件名和目录名统一使用 kebab-case。

## Skill 资产结构

```
knowledge-manager/
├── SKILL.md                              # 流程定义（知识更新 + 归档两套独立流程）
└── references/
    ├── initialization.md                 # 首次使用时的初始化动作（挂入口、建索引）
    ├── readme-template.md                # knowledge/README.md 的初始模板
    ├── writing-guidelines.md             # 技术栈无关的知识写作判断标准
    └── frontend-backend-collection.md    # 前后端分离场景专用的后台知识采集模板
```

各文件按需加载，详见 `SKILL.md` 的步骤编排。想深入理解写作判断力看 `references/writing-guidelines.md`——README 里讲的核心理念，里面有更完整的展开。

## 常见问题

### 1. 为什么不做异步，而是同步执行这个 Skill？

异步交给 subagent 看似更省当前会话的资源，但有三个绕不开的问题：

- **拿不到原始上下文**：异步的 subagent 无法直接获取主 Agent 的原始上下文。而如果靠不同工具（CodeBuddy / Claude 等）把会话历史持久化到磁盘的特性来传递，就得针对每种工具分别适配，方案会变得过于复杂。
- **双倍 Token 支出**：让 subagent 重新读一遍完整上下文，等于把 Token 花两遍。而在原始上下文里继续对话，前面的内容可以走缓存复用，消耗 Token 的费用比开全新会话的相差可以有 10 倍。
- **产物无法即时迭代**：你想对它生成的产物即时修改调整，就得能跟这个 subagent 继续对话。但当前 subagent 的设计做不到，只能另起一个会话处理——而那同样丢失了原始上下文。

为此目前采用同步方式，并接受它带来的两个缺点：

1. 会占用当前任务的时间。
2. 会占用当前会话的部分上下文空间。

当你只在任务结束或关键时刻才触发它时，相比上面的优点，这两个缺点是比较好接受的。
